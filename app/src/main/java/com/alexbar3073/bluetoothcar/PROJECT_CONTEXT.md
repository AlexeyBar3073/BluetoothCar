# 🚗 BluetoothCar Project - Контекст разработки

## 📱 О ПРОЕКТЕ

Приложение для отображения данных бортового компьютера автомобиля через Bluetooth соединение.
Получает данные от БК в формате JSON и отображает их в виджетах на экране.

Отлично, выявили важную проблему! Это типичный сценарий для Bluetooth приложений.

Проблема:
Устройство перезапускается (обновление прошивки) → соединение рвется
Программа не пытается восстановить соединение автоматически

Почему это происходит:
BluetoothService переходит в состояние DISCONNECTED или ERROR
Нет логики переподключения - после разрыва нужно пытаться переподключиться
Нет обработки разрыва соединения на уровне протокола

Решения:
1. Немедленное решение - добавить автоматическое переподключение в BluetoothService
2. Оптимальное решение - реализовать механизм реконнекта с экспоненциальной задержкой
3. Расширенное решение - добавить мониторинг состояния соединения и heartbeat

Предлагаю начать с немедленного решения:
Нужно обновить BluetoothService.kt чтобы:
При разрыве соединения (любой причине) пытаться переподключиться
Сохранять адрес устройства для переподключения
Иметь настраиваемые интервалы попыток

Что нужно изменить:
В BluetoothService добавить:
Сохранение deviceAddress и deviceName
Метод attemptReconnect()
Логику повторных попыток с таймаутами
В SharedViewModel обновить логику мониторинга состояния

Вопрос:
Хотите чтобы я сначала реализовал базовое переподключение или создал более надежный механизм с экспоненциальной задержкой?

Базовое решение (быстрое):
Простые повторные попытки каждые 5 секунд
Максимум 5 попыток

Продвинутое решение (более надежное):
Экспоненциальная задержка: 2s, 4s, 8s, 16s...
Учет успешных/неуспешных попыток
Остановка при выходе из приложения

Какой вариант предпочитаете?

Текущее состояние:
При запуске программа подключается к устройству, соединяется и получает данные.
При разрыве соединения тут же это инспектирует и старается подключиться заново.
Но возникает проблема с переподключением, видимо не отслеживается доступность устройства.
При текущей реализации получается что устройство должно быть обязательно включено в момент запуска программы.
Если сначала запустить программу, то она выдаст сообщение что не удалось подключиться к устройству.
И даже если после этого запустить устройство подключения не произойдет.
Это не верное поведение.

То есть при разрывае мы должны встать на прослушивание доступности нужного нам устройства.

Вернее так: при запуске программы мы из настройек получаем сохраненое устройство и запускаем механизм прослушивания доступности устройства.
Если устройство доступно, тогда начинаем попытки соединения с ним, с периодичностью 2 сек с течении 60 секунд (30 попыток)
Даже этого в принципе много. Если устройство доступно, но не соединяется, значит с ним есть проблемы и продолжать нет смысла.
Как только соединение произошло мы отправляем ему настройки, получаем ответ и запрашиваем данные. Подключаем поссавный слушатель данных.
При этом наш слушатель состояния соединения асинхронно находится в работе.
Как только он увидит что соединение пропало он устанавливает специальный флаг и мы перестаем слушать потоковые данные.
Как только слушатель увидит что устройство снова доступно, то начинаем все сначала: пытаемся соединиться, обмениваемся данными, пассивно прослушиваем поток.
И все тоже самое даже если у нас в настройках нет выбранного устройства при запуске программы.
Мы заходим в настройки, выбираем устройство, оно записывается в настройки, оповещает что есть устройство, мы читаем его из памяти и подключаем слушатель состояния...
Дальше все по цепочке.

То есть все должно работать просто и четко. Никаких хитрых схем не нужно.

ТЕХНИЧЕСКОЕ ОПИСАНИЕ АРХИТЕКТУРЫ BLUETOOTH CAR
ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ
Система Bluetooth Car представляет собой Android-приложение для взаимодействия с бортовым компьютером автомобиля через Bluetooth. Архитектура построена на принципах четкого разделения ответственности, реактивности и простоты контроля.

СЛОИ АРХИТЕКТУРЫ
1. UI СЛОЙ (ПРЕДСТАВЛЕНИЕ)
Компоненты: HomeScreen, SettingsScreen, виджеты
Ответственность: Отображение данных, обработка пользовательского ввода
Принцип: Пассивное отображение, минимальная логика

2. ViewModel СЛОЙ (ПОСРЕДНИК)
Компонент: SharedViewModel
Ответственность:
Подписка на потоки данных из бизнес-логики
Преобразование данных для UI
Передача пользовательских действий в бизнес-логику
Принцип: ТОЛЬКО посредник, НЕТ бизнес-логики

3. БИЗНЕС-ЛОГИКА (КООРДИНАЦИЯ)
Главный координатор: BluetoothConnectionManager
Ответственность:
Координация всех процессов подключения и обмена
Управление жизненным циклом подключения
Установка и контроль системных статусов
Делегирование задач специализированным помощникам

4. СПЕЦИАЛИЗИРОВАННЫЕ ПОМОЩНИКИ
DeviceAvailabilityMonitor: Поиск физического присутствия устройства
ConnectionStateManager: Установка и мониторинг физического соединения
DataStreamHandler: Управление протоколом обмена данными

5. СЕРВИСНЫЙ СЛОЙ (ИНФРАСТРУКТУРА)
BluetoothService: Предоставление низкоуровневого Bluetooth API
SettingsRepository: Управление настройками приложения
JsonParser: Обработка JSON сообщений

ПОТОК РАБОТЫ СИСТЕМЫ
ФАЗА 1: ИНИЦИАЛИЗАЦИЯ
Система стартует со статусом UNDEFINED
Загружаются настройки из SettingsRepository
Если сохраненное устройство найдено → переход к Фазе 2
Если устройство не выбрано → статус NO_DEVICE_SELECTED

ФАЗА 2: ПОДГОТОВКА К ПОДКЛЮЧЕНИЮ
BluetoothConnectionManager проверяет:
Поддержку Bluetooth на устройстве
Включен ли Bluetooth
Присутствие устройства в списке спаренных
При успехе → статус DEVICE_SELECTED

ФАЗА 3: ПОИСК УСТРОЙСТВА
Статус → SEARCHING_DEVICE
Запускается DeviceAvailabilityMonitor

Результаты:
Устройство найдено → колбек DEVICE_AVAILABLE
Устройство не найдено → колбек DEVICE_UNAVAILABLE (ожидание ручного запуска)

ФАЗА 4: УСТАНОВКА СОЕДИНЕНИЯ
Статус → CONNECTING
ConnectionStateManager выполняет физическое подключение (5 попыток)
Результаты:
Успех → колбек CONNECTED
Неудача → колбек DISCONNECTED (возврат к Фазе 3)

ФАЗА 5: МОНИТОРИНГ СОЕДИНЕНИЯ
ConnectionStateManager начинает постоянный мониторинг
При обнаружении разрыва → колбек DISCONNECTED (возврат к Фазе 3)

ФАЗА 6: ОБМЕН ДАННЫМИ
Запускается DataStreamHandler
Последовательность протокола:
Отправка настроек → статус SENDING_SETTINGS
Подтверждение настроек
Запрос данных → статус REQUESTING_DATA
Подтверждение запроса
Прием данных → статус LISTENING_DATA

Полученные данные передаются через BluetoothConnectionManager → SharedViewModel → UI

ПРИНЦИПЫ ВЗАИМОДЕЙСТВИЯ
КОЛБЕЧНАЯ АРХИТЕКТУРА
Все помощники общаются с BluetoothConnectionManager через колбеки

BluetoothConnectionManager является единственным подписчиком на все колбеки

Каждый помощник завершает работу после выполнения задачи

ПАССИВНЫЙ DATASTREAMHANDLER
Не выполняет активных опросов устройства

Подписывается на поток данных от BluetoothService

Отправляет настройки только при необходимости (подключение, изменение настроек)

BLUETOOTHSERVICE КАК ИНФРАСТРУКТУРА
Предоставляет только низкоуровневые Bluetooth операции
Не содержит бизнес-логики
Не управляет состояниями подключения
Все проверки разрешений и совместимости

КЛЮЧЕВЫЕ ТРЕБОВАНИЯ
Один координатор: BluetoothConnectionManager - единственный управляющий
Четкие границы: Каждый компонент делает ТОЛЬКО свою работу
Пассивный прием данных: Никаких активных опросов устройства

Реактивность: Использование Flow/StateFlow для передачи данных
Простота контроля: Возможность отследить каждый шаг через логи

Масштабируемость: Легкое добавление новых компонентов

Прозрачность: Легко понять, какой компонент что делает

Контролируемость: Каждый шаг можно отследить через логи

Надежность: Четкое разделение ответственности уменьшает ошибки

Тестируемость: Компоненты изолированы, легко мокируются

Масштабируемость: Новые функции добавляются как новые помощники

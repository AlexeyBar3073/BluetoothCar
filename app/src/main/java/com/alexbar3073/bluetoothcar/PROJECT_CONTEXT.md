## 2. Фаза 1: Инициализация приложения

### 2.1. Запуск MainActivity
- **Действие:** Пользователь запускает приложение
- **Инициатор:** Система Android
- **Результат:** Создается экземпляр `MainActivity`

### 2.2. Инициализация CoreModule
- **Действие:** `MainActivity` создает и инициализирует `CoreModule`
- **Назначение:** Синхронизация последовательности запуска всех компонентов приложения
- **Результат:** `CoreModule` готов к созданию `AppController`

### 2.3. Создание AppController
- **Действие:** `CoreModule` создает экземпляр `AppController`
- **Что происходит в init-блоке AppController:**
    1. Загрузка настроек из `SettingsRepository` → `_appSettings.value = settings`
    2. Конфигурация логгера (`AppLogger.configure(...)`)
    3. Ленивое создание `BluetoothConnectionManager` (при первом обращении)
    4. Вызов `bluetoothConnectionManager.updateSettings(settings)`
    5. Установка `_isInitialized.value = true`

### 2.4. Инициализация BluetoothConnectionManager (BCM)
- **Действие:** При первом обращении к `bluetoothConnectionManager` создается его экземпляр
- **Что происходит в init-блоке BCM:**
    1. Установка `_connectionStateFlow.value = ConnectionState.UNDEFINED`
    2. Вызов `createAllHelpers()`:
        - Создание `BluetoothService` (обертка над системным API)
        - Создание `ConnectionFeasibilityChecker`
        - Создание `DeviceAvailabilityMonitor`
        - Создание `ConnectionStateManager`
        - Создание `DataStreamHandler`
    3. Вызов `startCollectingCarData()`:
        - Подписка на `dataStreamHandler.carDataFlow`
        - Транзитная эмиссия данных в `_carDataFlow` (для `AppController`)

## 3. Фаза 2: Передача настроек и запуск процесса подключения

### 3.1. AppController передает настройки в BCM
- **Действие:** Вызов `bluetoothConnectionManager.updateSettings(settings)`
- **Входные данные:** Объект `AppSettings`, содержащий `selectedDevice` (БК) и другие параметры

### 3.2. BCM обрабатывает новые настройки
Логика BCM в методе `updateSettings()`:
1. Сохранение `newSettings` в `currentAppSettings`
2. Извлечение `newDeviceData = newSettings.selectedDevice`
3. Сравнение `currentBluetoothDeviceData?.address` с `newDeviceData?.address`
4. Обновление `currentBluetoothDeviceData = newDeviceData`
5. Ветвление по результату сравнения:

| Условие | Действие |
| :--- | :--- |
| Устройство изменилось (включая null) | Вызов `startConnectionProcess()` (запуск с первого шага) |
| Устройство НЕ изменилось И `isConnected = true` | Вызов `dataStreamHandler.updateAppSettings(newSettings)` |
| Устройство НЕ изменилось И `isConnected = false` | Ничего не делать (настройки сохранены) |

## 4. Фаза 3: Процесс подключения (пошагово)

### 4.1. BCM запускает процесс подключения
- **Метод:** `startConnectionProcess()`
- **Действия:**
    1. `stopAllProcesses()` — остановка всех помощников
    2. `feasibilityChecker.start(currentBluetoothDeviceData)`

### 4.2. Шаг 1: ConnectionFeasibilityChecker (CFC)

#### 4.2.1. Последовательность проверок CFC

| № | Проверка | Условие неудачи | Состояние при неудаче | Дальнейшие действия |
| :- | :--- | :--- | :--- | :--- |
| 1 | Устройство указано | `deviceData == null` или `address.isBlank()` | `NO_DEVICE_SELECTED` | Выход из `performChecks()` |
| 2 | Bluetooth адаптер доступен | `!bluetoothService.bluetoothAdapterIsAvailable()` | `ERROR` | Выход из `performChecks()` |
| 3 | Запуск мониторинга | — | — | Вызов `startBluetoothMonitoring()` |
| 4 | Bluetooth адаптер включен | `!bluetoothService.bluetoothAdapterIsEnabled()` | `BLUETOOTH_DISABLED` | Выход из `performChecks()` |
| 5 | Устройство сопряжено | `!isDevicePaired(deviceData)` | `NO_DEVICE_SELECTED` | Выход из `performChecks()` |
| 6 | Все проверки пройдены | — | `DEVICE_SELECTED` | Успешное завершение |

#### 4.2.2. Механизм мониторинга Bluetooth в CFC
Запуск мониторинга (`startBluetoothMonitoring()`):

```kotlin
bluetoothMonitoringDisposable = bluetoothService.monitorBluetoothState { isEnabled ->
    if (isEnabled) {
        // Bluetooth включился → вызываем DEVICE_SELECTED
        stateChangeCallback(ConnectionState.DEVICE_SELECTED, null)
    } else {
        // Bluetooth отключился → вызываем DISCONNECTED
        stateChangeCallback(ConnectionState.DISCONNECTED, "Bluetooth отключен")
    }
}
```

### 4.4. Шаг 2: DeviceAvailabilityMonitor (DAM)

#### 4.4.1. Конфигурация поиска

| Параметр | Значение |
| :--- | :--- |
| `MAX_SEARCH_ATTEMPTS` | 3 попытки |
| `RETRY_INTERVAL_MS` | 1000 мс (1 секунда) |

#### 4.4.2. Последовательность действий DAM
1. Уведомление о начале поиска: `stateChangeCallback(ConnectionState.SEARCHING_DEVICE, null)`
2. Цикл попыток поиска (до 3 раз): вызов `bluetoothService.startDiscovery()`

**Обработка найденного устройства (callback `onDeviceFound`):**
```kotlin
private val onDeviceFound: (String) -> Unit = { address ->
    if (address == targetAddress) {
        bluetoothService.stopDiscovery()
        stateChangeCallback(ConnectionState.DEVICE_AVAILABLE, null)
        stop()  // Остановка DAM
    }
}
```

**Обработка завершения поиска (callback `onDiscoveryFinished`):**
```kotlin
private val onDiscoveryFinished: () -> Unit = {
    if (isActive) {
        currentAttempt++
        if (currentAttempt < MAX_SEARCH_ATTEMPTS) {
            delay(RETRY_INTERVAL_MS)
            startDiscovery()
        } else {
            stateChangeCallback(ConnectionState.DEVICE_UNAVAILABLE, null)
            stop()
        }
    }
}
```

### 4.6. Шаг 3: ConnectionStateManager (CSM)

#### 4.6.1. Конфигурация подключения

| Параметр | Значение |
| :--- | :--- |
| `MAX_CONNECTION_ATTEMPTS` | 5 попыток |
| `CONNECTION_TIMEOUT_MS` | 10000 мс (10 секунд) |
| `RETRY_DELAY_MS` | 2000 мс (2 секунды) |

#### 4.6.2. Последовательность действий CSM
1. Уведомление: `stateChangeCallback(ConnectionState.CONNECTING, null)`
2. Цикл попыток подключения (до 5 раз)

```kotlin
private suspend fun performConnectionAttempts(deviceData: BluetoothDeviceData) {
    var attempts = 0
    while (attempts < MAX_CONNECTION_ATTEMPTS && isAttemptingConnection) {
        attempts++
        val (success, errorMessage) = tryConnectToDevice(deviceData)
        if (success) return
        if (attempts < MAX_CONNECTION_ATTEMPTS) {
            delay(RETRY_DELAY_MS)
        }
    }
    stateChangeCallback(ConnectionState.DISCONNECTED, null)
}
```

**Мониторинг соединения:**
```kotlin
private fun startConnectionMonitoring(deviceData: BluetoothDeviceData) {
    bluetoothService.startConnectionMonitoring(deviceData) { isConnected, eventDeviceData ->
        if (!isConnected) {
            stateChangeCallback(ConnectionState.DISCONNECTED, null)
            stopConnectionMonitoring()
        }
    }
}
```

#### 5.2.4. Обработка входящих данных (`handleIncomingData`)
```kotlin
private fun handleIncomingData(data: String) {
    val jsonObject = json.parseToJsonElement(data).jsonObject
    when {
        jsonObject.containsKey("settings") -> {
            settingsAcknowledged.set(true)
        }
        jsonObject.containsKey("GET_DATA") -> {
            commandAcknowledged.set(true)
        }
        jsonObject.containsKey("data") -> {
            val carData = json.decodeFromString<CarData>(dataElement.toString())
            _carDataFlow.emit(carData)
        }
    }
}
```

## 6. Фаза 5: Обработка разрыва соединения

### 6.1. BCM получает DISCONNECTED
**Источники DISCONNECTED:**
- От CFC: при отключении Bluetooth адаптера
- От CSM: при неудаче подключения или системном отключении
- От DSH: при ошибке протокола

**Действия BCM:**
1. Установка `isConnected = false`
2. Перезапуск процесса подключения: `startConnectionProcess()`

## 9. Таблица состояний и их значения

| Состояние | Значение | От кого приходит | Действие BCM |
| :--- | :--- | :--- | :--- |
| `UNDEFINED` | Начальное состояние | BCM инициализация | Ожидание |
| `NO_DEVICE_SELECTED` | Устройство не выбрано | CFC | Ожидание выбора |
| `BLUETOOTH_DISABLED` | Bluetooth выключен | CFC | Ожидание включения |
| `SEARCHING_DEVICE` | Поиск устройства | DAM | Ожидание результата |
| `DEVICE_AVAILABLE` | Устройство найдено | DAM | К CSM |
| `DEVICE_SELECTED` | Устройство прошло CFC | CFC | К DAM |
| `CONNECTING` | Установка соединения | CSM | Ожидание результата |
| `CONNECTED` | Соединение установлено | CSM | К DSH |
| `LISTENING_DATA` | Штатный прием данных | DSH | Мониторинг |
| `DISCONNECTED` | Соединение разорвано | CFC/CSM/DSH | Перезапуск (шаг 1) |
| `ERROR` | Критическая ошибка | Любой | Логирование |

# 🚗 BluetoothCar Project - Полный контекст разработки

## 📱 О ПРОЕКТЕ
Приложение для отображения данных бортового компьютера (БК) автомобиля через Bluetooth соединение. 
Система получает технические параметры от БК в формате JSON и визуализирует их с помощью кастомных графических приборов (Compose Canvas).
Поддерживаемая версия Android: 7.0 (API 24) и выше.

---

## 🏗️ АРХИТЕКТУРА СИСТЕМЫ
Архитектура построена на принципах Single Responsibility, реактивности и жесткой иерархии управления.

### 1. Слой бизнес-логики (Координация)
*   **AppController.kt**: Инициализирует репозитории, BCM, управляет жизненным циклом приложения и синхронизацией настроек.
*   **BluetoothConnectionManager (BCM)**: Единственный управляющий координатор связи. Делегирует задачи специализированным помощникам.

### 2. Специализированные помощники BCM (Listeners)
*   **ConnectionFeasibilityChecker**: Проверка разрешений, наличия BT-адаптера и его состояния.
*   **DeviceAvailabilityMonitor**: Поиск физического присутствия устройства в эфире.
*   **ConnectionStateManager**: Установка и постоянный мониторинг физического соединения (Socket).
*   **DataStreamHandler**: Управление логическим протоколом обмена данными (JSON).

### 3. Слой данных и инфраструктура
*   **BluetoothService.kt**: Низкоуровневая обертка над Android Bluetooth API. Только примитивные операции (connect, read, write).
*   **SettingsRepository.kt**: Хранение настроек через Jetpack DataStore.
*   **JsonParser.kt**: Обработка JSON сообщений (Settings, Command, Data).
*   **BluetoothLogger.kt**: Централизованное логирование событий системы.

### 4. Слой представления (UI)
*   **SharedViewModel.kt**: Единственный источник истины для UI. Транслирует данные от BCM в StateFlow.
*   **HomeScreen.kt**: Контейнер для дашбордов.
*   **Виджеты**: Набор кастомных компонентов на Canvas.

## 📡 ПРОТОКОЛ ОБМЕНА ДАННЫМИ
Обмен происходит в 5 фаз после установления физического соединения:

1.  **ФАЗА 1 (Настройки)**: Приложение отправляет JSON с параметрами авто каждые 1000мс.
2.  **ФАЗА 2 (Подтверждение)**: Ожидание от БК ответа `{"settings": "OK"}`.
3.  **ФАЗА 3 (Запрос)**: Отправка команды `{"command": "GET_DATA"}` каждые 1000мс.
4.  **ФАЗА 4 (Подтверждение)**: Ожидание от БК ответа `{"get_data": "OK"}`.
5.  **ФАЗА 5 (Поток)**: Переход в режим прослушивания. БК шлет данные `{"data": {...}}` по мере их изменения.

**Реконнект**: При разрыве система возвращается к Фазе 3 (поиск устройства) и повторяет цикл.

## 🏁 ТЕКУЩИЕ ПАРАМЕТРЫ ДАШБОРДА 4
Актуальные настройки визуализации основного комбинированного прибора:

*   **Иконки шкал**: Размер **20 ед.**, отступ от рисок **16 ед.**
*   **Нижние индикаторы**:
    *   **Топливо**: 20 ед.
    *   **Чек**: 24.5 ед.
    *   **Шины**: 22 ед.
*   **Позиционирование**: Иконки стоят **строго вертикально** (без вращения по дуге).
*   **Свечение (Glow)**: Радиус 0.9 от размера иконки, alpha 0.4.

## 🛠️ ТЕХНИЧЕСКИЕ ПРАВИЛА ОТРИСОВКИ
1.  **Сглаживание**: Использование `rotate` системы координат для предотвращения артефактов.
2.  **Направления**: Слева/сверху -> рост по часовой. Справа/снизу -> рост против часовой.
3.  **Единицы**: Все размеры привязаны к `geometry.unit` для адаптивности под разные экраны.

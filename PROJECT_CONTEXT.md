# 🚗 BluetoothCar Project - Полный контекст разработки

## 📱 О ПРОЕКТЕ
Приложение для отображения данных бортового компьютера (БК) автомобиля через Bluetooth соединение.
Система получает технические параметры от БК в формате JSON и визуализирует их с помощью кастомных графических приборов (Compose Canvas).
Поддерживаемая версия Android: 7.0 (API 24) и выше.

---

## 🏗️ АРХИТЕКТУРА И ЛОГИКА ВЗАИМОДЕЙСТВИЯ

Архитектура построена на принципах **Single Responsibility**, реактивности и жесткой иерархии управления:
`UI` ↔ `SharedViewModel` ↔ `AppController` ↔ `BluetoothConnectionManager` ↔ `Listeners` ↔ `BluetoothService`.

### 1. Главный координатор (The Mind) — `AppController.kt`
Центральный узел бизнес-логики. Живет на протяжении всего времени работы приложения.
*   **Инициализация**: При запуске загружает настройки из `SettingsRepository` и создает `BluetoothConnectionManager`.
*   **Управление состоянием**: Хранит `StateFlow` с настройками и данными БК для всего приложения.
*   **Синхронизация**: При изменении настроек в UI, сохраняет их в репозиторий и одновременно уведомляет `BCM`.
*   **Безопасность**: Гарантирует "чистую" остановку всех Bluetooth-процессов при выходе через метод `cleanup()`.

### 2. Посредник (The Bridge) — `SharedViewModel.kt`
Единственная точка контакта для UI.
*   **Пассивность**: Не содержит бизнес-логики. Только транслирует вызовы в `AppController` и предоставляет данные в формате, удобном для Compose.
*   **Реактивность**: Подписывается на потоки данных из `AppController` и предоставляет их UI через `StateFlow`.

### 3. Оркестратор связи — `BluetoothConnectionManager (BCM)`
Автономный модуль, управляющий жизненным циклом подключения.
*   **Логика принятия решений**: Получает `AppSettings`, сравнивает новое устройство со старым. Если устройство изменилось — полностью перезапускает цикл подключения. Если изменились только параметры (напр. объем бака) при активном соединении — командует `DataStreamHandler` отправить новые настройки.
*   **Управление помощниками**: Координирует 4 специализированных класса (Listeners).

### 4. Специализированные помощники (The Listeners)
Каждый помощник делает одну задачу и сообщает о результате через callback в BCM:

1.  **`ConnectionFeasibilityChecker`**: Шаг 1. Проверяет разрешения, наличие BT-адаптера, сопряжение устройства и включен ли Bluetooth. Запускает мониторинг состояния адаптера (если пользователь выключит BT в процессе — система это узнает).
2.  **`DeviceAvailabilityMonitor`**: Шаг 2. Ищет устройство в эфире (Discovery). Делает 3 попытки поиска с интервалом в 1 секунду.
3.  **`ConnectionStateManager`**: Шаг 3. Устанавливает физическое Socket-соединение. Делает до 5 попыток. После успеха запускает пассивный мониторинг разрыва связи.
4.  **`DataStreamHandler`**: Шаг 4. Реализует логический протокол обмена данными (JSON). Управляет очередностью: Настройки → Команда GET_DATA → Прием потока данных.

### 5. Инфраструктура
*   **`BluetoothService.kt`**: Низкоуровневая обертка. Только "грязные" операции с Android Bluetooth API.
*   **`SettingsRepository.kt`**: Jetpack DataStore. Обеспечивает персистентность настроек.
*   **`AppLogger.kt`**: Единая система логирования с тегами и временем для отладки протокола.

## 📡 ПРОТОКОЛ ОБМЕНА ДАННЫМИ

После установления физического соединения `DataStreamHandler` запускает следующую последовательность:

1.  **ФАЗА 1 (Настройки)**: Отправка JSON с параметрами (бак, форсунки и т.д.) каждые 1500мс.
    *   *Формат:* `{"settings": {"fuel_tank_capacity": 60, ...}}`
2.  **ФАЗА 2 (Подтверждение настроек)**: Ожидание от БК ответа, содержащего ключ `"settings"`.
3.  **ФАЗА 3 (Запрос данных)**: Только после Фазы 2. Отправка команды `{"command": "GET_DATA"}` каждые 1500мс.
4.  **ФАЗА 4 (Подтверждение команды)**: Ожидание от БК ответа с ключом `"GET_DATA"`.
5.  **ФАЗА 5 (Поток данных)**: Переход в режим прослушивания. БК шлет данные по мере изменения.
    *   *Формат:* `{"data": {"speed": 80.5, "fuel": 45.2, ...}}`

**Реконнект**: При любом разрыве (физическом или выключении BT) система через `BCM` возвращается на этап `UNDEFINED` и автоматически начинает цикл заново с поиска устройства.

## 🏁 ДАШБОРДЫ И ПРАВИЛА ОТРИСОВКИ

### Технические правила (Canvas)
1.  **Единицы**: Все размеры привязаны к `geometry.unit` (относительно размера контейнера) для адаптивности.
2.  **Направления**: Слева/сверху -> рост по часовой. Справа/снизу -> рост против часовой.
3.  **Сглаживание**: Использование `rotate` системы координат вместо ручного расчета тригонометрии там, где это возможно.

### Актуальные параметры (Dashboard Type 4)
*   **Иконки шкал**: Размер 20 ед., отступ 16 ед.
*   **Нижние индикаторы**: Топливо (20), Чек (24.5), Шины (22).
*   **Эффекты**: Glow (свечение) с радиусом 0.9 от размера иконки и alpha 0.4.

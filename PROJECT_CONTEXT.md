# 🚗 BluetoothCar Project - Полный контекст разработки

## 📱 О ПРОЕКТЕ
Приложение для отображения данных бортового компьютера (БК) автомобиля через Bluetooth соединение.
Система получает технические параметры от БК в формате JSON и визуализирует их с помощью кастомных графических приборов (Compose Canvas).
Поддерживаемая версия Android: 7.0 (API 24) и выше.

---

## 🏗️ АРХИТЕКТУРА И ЛОГИКА ВЗАИМОДЕЙСТВИЯ

Архитектура построена на принципах **Single Responsibility**, реактивности и жесткой иерархии управления:
`UI` ↔ `SharedViewModel` ↔ `AppController` ↔ `BluetoothConnectionManager` ↔ `Listeners` ↔ `AppBluetoothService`.

## 2. Фаза 1: Инициализация приложения

### 2.1. Запуск MainActivity
- **Действие:** Пользователь запускает приложение.
- **Инициатор:** Система Android.
- **Результат:** Создается экземпляр `MainActivity`.

### 2.2. Инициализация CoreModule
- **Действие:** `MainActivity` создает и инициализирует `CoreModule`.
- **Назначение:** Синхронизация последовательности запуска всех компонентов приложения.
- **Результат:** `CoreModule` готов к созданию `AppController`.

### 2.3. Создание AppController
- **Действие:** `CoreModule` создает экземпляр `AppController`.
- **Что происходит в init-блоке AppController:**
    1. Загрузка настроек из `SettingsRepository` → `_appSettings.value = settings`.
    2. Конфигурация логгера (`AppLogger.configure(...)`).
    3. Инициализация БД ошибок ЭБУ и комбинаций из JSON (ecu_errors.json, ecu_combinations.json).
    4. Ленивое создание `BluetoothConnectionManager` (при первом обращении).
    5. Вызов `bluetoothConnectionManager.updateSettings(settings)`.
    6. Установка `_isInitialized.value = true`.

### 2.4. Инициализация BluetoothConnectionManager (BCM)
- **Действие:** При первом обращении к `bluetoothConnectionManager` создается его экземпляр.
- **Что происходит в init-блоке BCM:**
    1. Установка `_connectionStateFlow.value = ConnectionState.UNDEFINED`.
    2. Вызов `createAllHelpers()`:
        - Создание 4 помощников (`CFC`, `DAM`, `CSM`, `DSH`) с внедрением диспетчера для тестов.
    3. Вызов `startCollectingCarData()`:
        - Подписка на `dataStreamHandler.carDataFlow`.
        - Транзитная эмиссия данных в `_carDataFlow` (для `AppController`).

## 3. Фаза 2: Передача настроек и запуск процесса подключения

### 3.1. Изменение настроек пользователем
При изменении настроек на форме (объем бака, форсунки, выбор БК и т.д.):
1. **UI (SettingsScreen)** вызывает метод обновления в `SharedViewModel`.
2. **SharedViewModel** транслирует вызов в `AppController.updateSettings(newSettings)`.
3. **AppController**:
   - Обновляет внутренний экземпляр `_appSettings`.
   - Сохраняет настройки в **SettingsRepository** (DataStore).
   - Передает настройки в **BCM** через `bluetoothConnectionManager.updateSettings(newSettings)`.

### 3.2. BCM обрабатывает новые настройки
Логика BCM в методе `updateSettings()`:
1. Сохранение `newSettings` в `currentAppSettings`.
2. Извлечение `newDeviceData = newSettings.selectedDevice`.
3. Сравнение `currentBluetoothDeviceData?.address` с `newDeviceData?.address`.
4. Обновление `currentBluetoothDeviceData = newDeviceData`.
5. Ветвление по результату сравнения:

| Условие | Действие |
| :--- | :--- |
| Устройство изменилось (включая null) | Вызов `startConnectionProcess()` (запуск с первого шага) |
| Устройство НЕ изменилось И `isConnected = true` | Вызов `dataStreamHandler.updateAppSettings(newSettings)` |
| Устройство НЕ изменилось И `isConnected = false` | Ничего не делать (настройки сохранены) |

## 4. Фаза 3: Процесс подключения (пошагово)

### 4.1. BCM запускает процесс подключения
- **Метод:** `startConnectionProcess()`
- **Действия:**
    1. `stopAllProcesses()` — остановка всех помощников.
    2. `feasibilityChecker.start(currentBluetoothDeviceData)`.

### 4.2. Шаг 1: ConnectionFeasibilityChecker (CFC)
Проверяет аппаратную и системную готовность.
1. **Устройство указано**: `deviceData != null` и адрес валиден.
2. **Адаптер доступен**: `bluetoothService.bluetoothAdapterIsAvailable()`.
3. **Запуск мониторинга**: `startBluetoothMonitoring()`. Мониторинг отслеживает системное включение/выключение Bluetooth.
4. **Адаптер включен**: `bluetoothService.bluetoothAdapterIsEnabled()`. Если выключен — состояние `BLUETOOTH_DISABLED`.
5. **Сопряжение**: `isDevicePaired(deviceData)`.
**Результат**: Состояние `DEVICE_SELECTED`.

### 4.3. Шаг 2: DeviceAvailabilityMonitor (DAM)
Поиск БК в Bluetooth-эфире (Discovery).
- **Попытки**: До 3 раз.
- **Интервал**: 1000 мс между попытками.
- **Результат**: Если устройство найдено по MAC-адресу — состояние `DEVICE_AVAILABLE`. Если попытки исчерпаны — `DEVICE_UNAVAILABLE`.

### 4.4. Шаг 3: ConnectionStateManager (CSM)
Установление физического Socket-соединения.
- **Попытки**: До 5 раз.
- **Таймаут**: 10000 мс на попытку.
- **Интервал**: 2000 мс между попытками.
- **Мониторинг**: После успеха запускает `startConnectionMonitoring` для отслеживания разрыва.
**Результат**: Состояние `CONNECTED`.

### 4.5. Шаг 4: DataStreamHandler (DSH)
Реализация логического протокола обмена данными (JSON).
1. **Подписка**: `startReceiver()` запускает прослушивание входящего потока от `AppBluetoothService`.
2. **Очередь команд**: Все исходящие пакеты (Настройки, GET_DATA, JSON) попадают в `commandQueue`.
3. **MsgID & AckID**: Каждая команда упаковывается в JSON-конверт с уникальным `msg_id`. Воркер ждет подтверждения с `ack_id` от БК.
4. **Реактивность**: Использование `withTimeoutOrNull` и `SharedFlow` для мгновенного пробуждения при получении подтверждения.
5. **Прием данных**: При получении первого валидного пакета `{"telemetry": {...}}` устанавливается статус `LISTENING_DATA`.

## 5. Фаза 5: Обработка разрыва соединения
При получении `DISCONNECTED` от любого помощника:
1. `isConnected = false`.
2. `BCM` вызывает `startConnectionProcess()`, возвращаясь на Шаг 1 (CFC).

---

## 6. Фаза 6: Ручное управление
- **retryConnection()**: Принудительный перезапуск цикла подключения с Шага 1.
- **disconnectFromDevice()**: Сброс `selectedDevice` в настройках, что триггерит остановку всех процессов.
- **sendJsonCommand()**: Прямая отправка команды в `DataStreamHandler`.
- **full_tank**: Команда установки отметки полного бака (ранее `reset_fuel`). На Dashboard 4 отправляется при долгом клике по центру комбинированного прибора в режиме FUEL.
- **reset_avg**: Команда сброса общего среднего расхода топлива. На Dashboard 4 (TripWidget) отправляется при долгом клике по значению AVG.
- **reset_avg_cur**: Команда сброса расхода топлива за текущую поездку. На Dashboard 4 (TripWidget) отправляется при долгом клике по значению T.AVG.

## 🏁 ДАШБОРДЫ И ПРАВИЛА ОТРИСОВКИ
1. **Адаптивность**: Все размеры привязаны к `geometry.unit`.
2. **Направления**: Слева/сверху -> рост по часовой. Справа/снизу -> рост против часовой.
3. **Canvas**: Использование `rotate` системы координат вместо ручного расчета тригонометрии.
4. **Эффекты**: Glow (свечение) с радиусом 0.9 от размера иконки и alpha 0.4.

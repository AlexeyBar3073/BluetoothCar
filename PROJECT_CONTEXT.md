# 🚗 BluetoothCar Project - Полный контекст разработки

## 📱 О ПРОЕКТЕ
Приложение для отображения данных бортового компьютера (БК) автомобиля через Bluetooth соединение. 
Система получает технические параметры от БК в формате JSON и визуализирует их с помощью кастомных графических приборов (Compose Canvas).

---

## 🏗️ АРХИТЕКТУРА СИСТЕМЫ
Архитектура построена на принципах Single Responsibility, реактивности и жесткой иерархии управления.

### 1. Слой бизнес-логики (Координация)
*   **AppController.kt**: "Мозг" системы. Инициализирует репозитории, BCM, управляет жизненным циклом приложения и синхронизацией настроек.
*   **BluetoothConnectionManager (BCM)**: Главный координатор связи. Он не делает работу сам, а управляет "помощниками".

### 2. Специализированные помощники BCM
*   **ConnectionFeasibilityChecker**: Проверка разрешений, наличия BT-адаптера и его включенного состояния.
*   **DeviceAvailabilityMonitor**: Сканирование эфира и проверка физической доступности выбранного устройства.
*   **ConnectionStateManager**: Управление физическим подключением (сокетом) и мониторинг разрывов.
*   **DataStreamHandler**: Управление логическим протоколом обмена (JSON).

### 3. Слой данных (Инфраструктура)
*   **BluetoothService.kt**: Низкоуровневая обертка над Android Bluetooth API. Только примитивные операции (connect, read, write).
*   **SettingsRepository.kt**: Хранение настроек через Jetpack DataStore.
*   **JsonParser.kt**: Сериализация и десериализация команд и данных.

### 4. Слой представления (UI)
*   **SharedViewModel.kt**: Единственный источник истины для UI. Получает данные от бизнес-логики и транслирует их в StateFlow.
*   **HomeScreen.kt**: Контейнер, переключающий типы дашбордов.
*   **Виджеты**: Набор кастомных UI-компонентов (спидометры, термометры и т.д.).

## 📡 ПРОТОКОЛ ОБМЕНА ДАННЫМИ
Обмен происходит в 5 фаз после установления физического соединения:

1.  **ФАЗА 1 (Настройки)**: Приложение отправляет JSON с параметрами авто (бак, форсунки, импульсы скорости) каждую секунду.
2.  **ФАЗА 2 (Подтверждение)**: Ожидание от БК ответа `{"settings": "OK"}`.
3.  **ФАЗА 3 (Запрос)**: Отправка команды `{"command": "GET_DATA"}` каждую секунду.
4.  **ФАЗА 4 (Подтверждение)**: Ожидание от БК ответа `{"get_data": "OK"}`.
5.  **ФАЗА 5 (Поток)**: Переход в режим прослушивания. БК шлет данные по мере их изменения.

## ✅ ТЕКУЩЕЕ СОСТОЯНИЕ И ЛОГИКА ЭЛЕМЕНТОВ
### Дашборд 4 (DashboardType4)
*   **Спидометр**: Имеет встроенный в нижнюю часть вольтметр.
*   **Вольтметр**: 
    *   Иконка (`ic_battery_full_50`) и текст значения (`12.4V`) расположены в одну горизонтальную линию.
    *   При низком напряжении (<= 11.8V) иконка становится ярко-алой (`#FF0000`) и мигает (alpha 0.4 - 1.0).
    *   Шкала вольтметра вращается в "окне" в нижней части прибора.
*   **Комбинированный прибор**: Позволяет переключать отображение (Топливо / Темп. АКПП / Темп. Двигателя) по тапу в центр или в соответствующие сектора.

## 🛠️ ТЕХНИЧЕСКИЕ ОСОБЕННОСТИ ОТРИСОВКИ
1.  **Сглаживание иконок**: Для предотвращения "лесенок" при повороте иконок на Canvas используется метод `rotate` всей системы координат относительно центра прибора, а не локальный поворот в точке.
2.  **Шкалы**: 
    *   Слева/сверху -> рост по часовой стрелке.
    *   Справа/снизу -> рост против часовой стрелки.
